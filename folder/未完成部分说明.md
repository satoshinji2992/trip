# 个性化旅游系统 - 当前限制和改进计划

> 本文件保留原文件名，便于课程材料连续管理。当前系统核心功能已完成，以下内容主要说明仍可继续增强的工程化和数据质量方向。

## 二、当前数据和运行限制

### 1. 数据来源限制

系统已满足课程数据量要求：

- 景区/校园：202个
- 建筑物：4380个，每个景区/校园不少于20个
- 服务设施：10100个，每个景区/校园50个，10种类型
- 道路边：44440条，每个景区/校园220条
- 用户：14个

当前为了满足课程“景区和校园内部可以一致”的要求，景区复用故宫道路模板，校园复用北京大学道路模板。系统代码支持本地GeoJSON和OpenStreetMap Overpass导入真实地图，但仓库中未保存大规模真实地图数据文件。

后续改进：

- 为更多景区/校园采集官方或OSM内部地图。
- 将建筑物、设施点位从模拟坐标替换为真实坐标。
- 引入地图数据版本管理，区分模板数据和真实采集数据。

### 2. AIGC服务限制

系统已接入Wan图生视频服务，后端通过：

```text
AIGC_BASE_URL
AIGC_ENDPOINT
```

配置视频生成服务器地址。没有 `.env` 文件时会使用 `config.py` 中的默认地址。

当前限制：

- 视频生成依赖外部GPU服务或局域网服务器。
- 生成耗时较长，同步请求会占用等待时间。
- 当前保存生成视频到本地 `backend/uploads/`。

后续改进：

- 使用异步任务队列处理视频生成。
- 增加生成进度查询。
- 对生成视频做云存储和CDN分发。

### 3. 存储和并发限制

当前数据库使用SQLite，适合课程演示和轻量部署。SQLite自带文件锁，能保证基本读写一致性，但高并发写入能力有限。

后续改进：

- 生产环境迁移到PostgreSQL或MySQL。
- 将全文索引和上传文件目录从本地磁盘迁移到独立服务。
- 对只读缓存增加显式锁或使用Redis缓存。

### 4. 媒体能力限制

当前支持：

- 日记图片上传和删除。
- AIGC生成视频展示。

尚未专门实现用户直接上传视频和视频转码。

后续改进：

- 增加用户原始视频上传。
- 接入FFmpeg做转码、封面截取和压缩。
- 增加视频时长、大小和格式校验。

### 5. 部署限制

已提供：

- `deploy/gunicorn.conf.py`
- `deploy/nginx.trip.conf`
- `folder/Nginx多Worker部署说明.md`

但当前未实际部署到公网服务器。

后续改进：

- 部署到云服务器。
- 配置HTTPS证书和域名。
- 增加日志轮转、监控和自动重启。

## 三、总结

系统已经覆盖课程要求中的核心数据、核心算法和核心功能。当前主要限制集中在真实数据采集、生产环境部署、AIGC异步化和云端存储等工程增强方向。这些限制不影响课程演示运行，但可以作为后续迭代和答辩反思内容。
