# 个性化旅游系统 - 评价和改进意见报告

## 一、总体评价

本系统围绕景区和校园游览场景，完成了旅游推荐、路线规划、场所查询、旅游日记、美食推荐、室内导航和AIGC旅游动画生成等核心功能。系统采用前后端分离架构，前端负责页面展示和交互，后端负责数据管理、业务接口和核心算法计算，数据库使用SQLite保存景区、校园、道路图、设施、日记和美食等数据。系统的主要特色包括微调并本地部署Wan2.2图生视频大模型、Nginx + Gunicorn一键部署、zlib压缩存储、BM25F全文检索和道路拥挤度路线规划。

从课程设计要求来看，系统已经满足主要数据规模和功能要求。当前数据库包含202个景区/校园、4380个建筑物、10100个服务设施、29692个道路节点、44440条道路边、14个系统用户、924个餐厅、4105道美食和26篇旅游日记。路线规划、推荐排序、场所查询、全文检索、日记压缩等功能均能对应课程要求中的核心算法点。

系统整体可运行性较好，提供了可执行发布包，并支持通过Nginx和Gunicorn进行多worker部署。对于课程验收，可以完整演示从目的地推荐、内部路线规划、附近设施查询，到日记记录、视频生成和美食推荐的完整流程。

## 三、算法和数据结构评价

系统在不同模块中选择了较合适的数据结构和算法：

1. 路线规划模块使用邻接表存储道路图，并使用最小堆优化Dijkstra算法，适合非负权道路图的最短路径计算。
2. 多点路线规划采用关键点最短路径矩阵、最近邻贪心和2-opt优化，避免直接枚举所有路径排列，兼顾效率和路线质量。
3. 推荐模块使用Top-K最小堆，只维护前K个结果，符合用户通常只关注前10个推荐项的使用场景。
4. 名称和关键词查询使用Trie前缀查找、编辑距离模糊匹配，提高查询容错能力。
5. 旅游日记正文使用Whoosh全文索引和BM25F评分，适合长文本检索。
6. 旅游日记正文使用zlib无损压缩，减少存储占用，同时保证内容可完整恢复。
7. 场所查询使用Dijkstra计算道路图上的实际可达距离，避免用直线距离造成结果不准确。

总体来看，系统没有把所有功能简单交给数据库排序或字符串包含查询，而是在业务层结合课程要求实现了有针对性的数据结构和算法。

## 四、系统优点

### 1. 功能链条完整

系统覆盖旅游前、旅游中、旅游后的主要场景。用户可以先选择目的地，再进行路线规划和附近设施查询，最后通过旅游日记记录游览过程，并查看美食推荐。功能之间衔接较自然。

### 2. 数据规模满足验收要求

系统数据量达到课程设计要求，景区/校园数量、建筑物数量、服务设施数量、道路边数量和用户数量均满足最低要求。每个景区/校园内部道路图都有足够的节点和边，可支撑路线规划与场所查询。

### 3. 路线规划结果更接近真实场景

路线规划不仅支持最短距离，还支持最短时间、道路拥挤度和交通工具限制。最短时间策略会根据道路拥挤度调整速度，因此能体现“近但拥挤”和“远但通畅”的差异。

### 4. 场所查询符合实际可达距离

附近设施不是按直线距离排序，而是按道路图上的最短路径距离排序。该设计更符合真实游览场景，也满足课程对场所查询算法的要求。

### 5. 具备可运行发布能力

项目提供release发布包，包含源码、数据库、启动脚本、环境配置和部署说明。通过 `setup_and_start_nginx.sh` 可以自动安装依赖、初始化数据、构建前端并启动Nginx + Gunicorn。

## 五、当前不足

### 1. 地图数据真实性仍可提升

当前系统支持本地GeoJSON和OpenStreetMap Overpass导入真实地图，但为了保证课程数据规模和稳定演示，景区主要复用故宫风格道路模板，校园主要复用北京大学风格道路模板。虽然满足“景区和校园内部可以一致”的要求，但真实细节仍不够丰富。

### 2. AIGC视频生成依赖外部服务

旅游动画生成依赖本地部署的Wan2.2图生视频服务。如果视频服务器未启动、GPU资源不足或网络地址配置不正确，前端无法完成视频生成。当前系统已经支持配置服务地址，但还没有实现异步任务队列和生成进度查询。

### 3. SQLite适合演示但不适合高并发生产环境

SQLite部署简单，适合课程设计和单机演示，但在高并发写入、多人同时上传图片或评论评分时，性能和锁竞争能力有限。

### 4. 媒体管理能力仍较基础

当前支持图片上传、删除和AIGC生成视频展示，但没有实现用户直接上传原始视频、视频转码、封面截取、在线播放质量切换等功能。

### 5. 测试覆盖仍可进一步系统化

当前测试主要以功能演示和手工联调为主，能够验证主要流程可运行，但自动化测试覆盖不足。对于路线规划、Top-K排序、模糊查询、压缩解压等核心算法，可继续补充单元测试和边界测试。

## 六、改进计划

### 1. 数据质量改进

- 为更多代表性景区和校园采集真实GeoJSON或OSM道路数据。
- 将建筑物、服务设施和餐厅坐标逐步替换为真实坐标。
- 为道路边补充更细的属性，例如路面类型、是否无障碍、开放时间等。
- 增加地图数据版本管理，区分模板数据、真实采集数据和测试数据。

### 2. 路线规划改进

- 对多点路线规划增加更多优化策略，例如模拟退火、遗传算法或动态规划小规模精确解。
- 增加路线偏好设置，例如少走楼梯、避开拥挤路段、优先经过服务设施等。
- 在前端地图中增加更清晰的路段说明、转向提示和距离标注。

### 3. AIGC功能改进

- 将视频生成改为异步任务，生成过程中返回任务ID。
- 增加视频生成进度查询和失败重试机制。
- 保存生成参数和历史记录，便于用户复用提示词。
- 对生成视频增加封面图、时长、分辨率等元数据展示。

### 4. 存储和部署改进

- 生产环境将SQLite迁移到MySQL或PostgreSQL，提高并发写入能力。
- 将上传图片和生成视频迁移到对象存储，减少本地磁盘压力。
- 增加日志轮转、异常监控和服务自动重启配置。
- 配置HTTPS和正式域名，提升公网访问安全性。

### 5. 测试和维护改进

- 为图算法、排序算法、搜索算法和压缩模块补充单元测试。
- 为推荐、路线规划、场所查询、日记和美食模块补充接口测试。
- 增加前端关键页面的端到端测试，覆盖登录、搜索、规划、上传和生成视频流程。
- 建立固定测试数据集，便于每次修改后验证功能一致性。

## 七、总结

本系统已经完成课程要求中的核心数据、核心功能和核心算法，具备完整演示能力和可运行发布包。系统的优势在于功能链条完整、算法点覆盖充分、路线规划和场所查询较贴近真实使用场景。后续改进重点应放在真实地图数据扩展、AIGC异步化、数据库升级、媒体管理和自动化测试上，从而进一步提升系统的稳定性、真实性和可维护性。
