# 个性化旅游系统课程设计报告

课程设计名称：个性化旅游系统的设计与实现

## 课程设计内容

本课程设计完成了一个面向景区与校园游览场景的个性化旅游系统。系统覆盖旅游前的景区/校园推荐、旅游中的路线规划、室内导航、场所查询，以及旅游后的日记交流、美食推荐和AIGC旅游动画生成。项目采用React + Vite前端、Flask + SQLAlchemy后端、SQLite数据库和Nginx + 多Gunicorn后端实例部署方案，并集成微调后本地部署的Wan2.2图生视频大模型。系统准备了202个景区/校园、4380个建筑、10100个服务设施、29692个道路节点、44440条道路边、14个用户、924个餐厅、4105道美食和26篇旅游日记，满足课程设计对数据规模和功能运行的要求。

## 一、系统创新性和实际价值

本系统围绕“旅游前推荐、旅游中导航、旅游后记录分享”设计，目标不是只做静态信息展示，而是把数据结构算法落到真实可操作的旅游流程中。

核心创新点如下表所示。

| 创新点 | 实现方式 | 实际价值 |
| --- | --- | --- |
| 微调并本地部署Wan2.2大模型 | 在 `wan2.2_traval_fintuned` 中提供Wan2.2图生视频服务，后端按图片、提示词、分辨率、帧数和seed调用生成接口 | 将普通旅游照片生成短视频，形成“文字+图片+视频”的旅游日记表达方式，增强系统展示效果 |
| Nginx + 多后端实例一键部署 | 发布包提供 `setup_and_start_nginx.sh`，自动安装依赖、生成环境配置、初始化数据库、构建前端并启动Nginx和多个Gunicorn后端实例 | 降低部署和复现成本，使系统更接近真实Web应用运行方式 |
| 日记压缩存储 | 日记正文写入数据库前使用zlib无损压缩，读取时自动解压 | 降低长文本存储占用，同时保证用户原文不丢失 |
| 日记全文检索 | 使用Whoosh建立索引，并采用BM25F进行相关性排序 | 支持按日记正文内容搜索，比简单标题查询更符合真实游记检索需求 |
| 道路拥挤度路线规划 | 道路边保存拥挤度，最短时间策略按拥挤度修正速度 | 使最短距离和最短时间可能产生不同路线，增强路径规划的真实性和可解释性 |
| 实际路径距离场所查询 | 基于道路图运行Dijkstra，按可达路径距离排序附近设施 | 避免直线距离误差，更符合用户在景区/校园内找设施的真实体验 |
| Top-K推荐排序 | 推荐、美食和日记模块使用容量为K的最小堆维护前K项 | 用户通常只看前10条结果，避免全量排序，提高排序效率 |

系统的实际价值在于把目的地选择、路线安排、附近设施查找、游记分享、美食发现整合在同一个平台中。用户在旅游前可以根据兴趣得到推荐，旅游中可以查看路线和附近服务设施，旅游后可以用文字、图片和视频记录经历，形成完整的闭环。

### 用户需求满足情况

## 二、系统架构和模块接口

系统采用前后端分离架构：前端使用React 18、Vite 5、Ant Design和TailwindCSS实现页面与交互；后端使用Flask 3.0提供RESTful API；数据层使用SQLAlchemy ORM连接SQLite数据库；部署层提供Nginx反向代理、least_conn负载均衡和多个Gunicorn后端实例运行配置。

后端按模型层、路由层、服务层组织。模型层包括用户、景区、建筑、道路图、设施、日记、餐厅和美食等模型；路由层包括auth、scenic、route_plan、facility、diary、food、navigation等蓝图；服务层包括graph_service、sort_service、search_service，集中实现图算法、排序算法和搜索算法。

主要接口包括：/api/scenic用于景区/校园列表、搜索和推荐；/api/route用于地图数据、单点路径和多点路径规划；/api/facility用于附近设施和类别过滤；/api/diary用于日记CRUD、评论评分、全文检索、图片上传和AIGC视频生成；/api/food用于美食推荐、菜系过滤和模糊查询；/api/navigation用于室内导航。前端通过Axios统一封装接口，开发环境由Vite代理到后端，部署环境由Nginx代理/api/和/uploads/。

## 三、系统开发模式

系统采用“前后端分离 + 分层模块化 + 数据驱动 + 算法服务层”的开发模式。前端负责页面展示、地图交互和表单操作，后端负责RESTful接口、数据库访问和核心算法计算；模型层、路由层、服务层相互分离，图算法、排序算法和搜索算法集中在services目录中，便于独立测试和复用。数据构建采用初始化脚本批量生成景区/校园、建筑、设施、道路图、餐厅、美食、用户和旅游日记，保证功能演示依赖稳定数据集。开发过程中使用测试agent辅助进行接口检查、异常定位和回归测试；发布阶段通过脚本生成可运行包，并使用Nginx + 多Gunicorn后端实例提高运行稳定性。

## 四、数据规模和数据字典概述

系统当前数据库满足课程设计的数据规模要求。景区和校园共202个，其中景区117个、校园85个；建筑物4380个；服务设施10100个，覆盖洗手间、超市、餐厅、咖啡馆、商店、医务室、停车场等10类；道路节点29692个，道路边44440条，每个景区/校园220条；系统用户14个；餐厅924个；美食4105道；旅游日记26篇并包含评论、图片和生成视频记录。

核心表包括users、scenics、buildings、facilities、graph_nodes、graph_edges、restaurants、foods、diaries、diary_comments、diary_images、indoor_nodes和indoor_edges。道路边表保存距离、道路名称、拥挤度、是否双向、可通行交通工具等字段；日记表使用content_compressed字段保存压缩后的正文；设施和餐厅通过graph_node_id绑定道路图节点，从而支持按实际路径距离排序。

真实地图方面，init_data_graph.py支持本地GeoJSON和OpenStreetMap Overpass导入，失败时自动回退到模拟道路图。当前为满足“景区和校园内部可以一致”的课程要求，景区复用故宫风格道路模板，校园复用北京大学风格道路模板，并保证每个内部地图边数不少于200条。

## 五、完成的主要功能

1. 旅游推荐：用户可以浏览景区和校园，根据类型、类别和关键词查询；系统可以按热度、评分和个人兴趣进行Top-K推荐。

2. 路线规划：支持单点最短路径、多点路线规划、返回起点、最短距离、最短时间、步行/自行车/电瓶车策略；道路按拥挤度可视化，规划结果展示距离、时间和平均拥挤度。

3. 室内导航：模拟教学楼、博物馆等建筑内部结构，支持大门、电梯、楼层、房间之间的路径规划和分步导航。

4. 场所查询：用户选中当前位置后，可以查找附近设施，支持类别过滤和关键词搜索，结果按道路图上的实际路径距离从近到远排序。

5. 旅游日记：支持日记创建、编辑、浏览、评论、评分、按目的地查询、标题查询、全文检索、zlib压缩存储、图片上传和删除。

6. AIGC旅游动画：用户可选择日记中的任意图片，修改英文提示词，设置横竖屏、480p/720p、时长、采样步数和seed，后端调用微调并本地部署的Wan2.2图生视频服务生成mp4并在详情页保留展示。

7. 美食推荐：选中景区/校园后，可按热度、评分、距离排序并按菜系过滤，也支持美食名、菜系、饭店或窗口名称的模糊查询。

8. 可执行发布包：scripts/build_release.sh生成release/trip-runnable.zip，包内包含源码、数据库、文档、启动脚本和Nginx/多Gunicorn后端实例部署脚本，支持一键部署运行。

## 六、算法设计、选择原因和性能分析

路线规划使用邻接表图和自实现最小堆优化Dijkstra算法。景区/校园道路距离和时间均为非负权重，Dijkstra能稳定求出单源最短路径，时间复杂度为O((V+E)logV)，适合本系统每个地图约220条边的规模。最短距离策略以distance作为权重；最短时间策略根据道路拥挤度和交通工具速度计算time_cost，因此能得到与最短距离不同的路线。

多点路线规划接近旅行商问题，若精确枚举所有访问顺序复杂度过高。系统先对起点和目标点运行Dijkstra得到关键点距离矩阵，再用最近邻贪心构建初始路线，最后用2-opt反转局部路径减少总成本。该方法不能保证全局最优，但响应快、效果稳定，适合交互式系统。

推荐排序使用Top-K最小堆，容量为K。遍历候选项时只维护当前分数最高的K个结果，复杂度为O(NlogK)，优于完整排序O(NlogN)。用户通常只看前10个景点、日记或美食，因此Top-K比全量排序更符合需求。综合推荐分数采用热度、评分和兴趣匹配加权，其中旅游推荐可采用0.4热度、0.3评分、0.3兴趣匹配。

查询算法分为短文本查询和长文本检索。景区、日记标题和名称类字段使用Trie前缀树，查询复杂度为O(L)；模糊查询使用自实现Levenshtein编辑距离，能处理用户输入不完整或有错字的情况，单次匹配复杂度为O(MN)；日记正文较长，因此使用Whoosh和BM25F全文检索，根据词频、文档长度和字段权重排序，比简单contains更适合游记内容搜索。

场所查询使用Dijkstra从当前位置计算到所有节点的最短路径距离，再对设施节点按路径距离排序。这样得到的是沿道路可达的实际距离，而不是直线距离，满足附近设施查询按真实路径距离排序的要求。日记正文使用zlib无损压缩，写入时压缩、读取时解压，能减少长文本存储占用且不损失内容。

### 算法选择理由与优缺点补充

本系统的算法选择以“满足用户真实操作、保证交互响应、便于解释”为原则。Dijkstra适合非负权道路图，优点是结果稳定、可同时支持距离和时间权重，缺点是遇到频繁变化的大规模路网时需要结合缓存或增量更新；多点路线采用贪心+2-opt近似，优点是响应快、实现清晰，缺点是不能保证全局最优；Top-K最小堆避免完整排序，适合只展示前若干推荐结果，缺点是需要先定义合理评分函数；Trie和编辑距离提升查询效率与容错能力，但模糊匹配在候选较多时仍需控制搜索范围；BM25F适合日记长文本检索，优点是相关性排序更自然，缺点是需要维护索引；zlib压缩能降低文本存储占用，代价是读写时增加少量压缩和解压开销。

## 八、系统性能和成本分析

计算性能方面，推荐和排序模块使用Top-K将排序复杂度从O(NlogN)降低到O(NlogK)；路径规划使用堆优化Dijkstra，适合中等规模景区/校园道路图；全文检索使用Whoosh索引避免对所有日记正文做低效扫描。

部署性能方面，开发环境可使用Flask和Vite方便调试，发布包提供Nginx + 多Gunicorn后端实例方案。Nginx负责前端静态资源、上传大小限制和反向代理，Nginx通过least_conn分发到多个Gunicorn后端实例，每个实例通过worker处理后端请求；setup_and_start_nginx.sh 支持一键安装依赖、初始化数据库、构建前端并启动服务。该方案比Flask开发服务器更适合多用户并发访问。

成本方面，SQLite无需独立数据库服务器，适合单机运行和轻量部署；图片和视频保存在本地uploads目录，部署成本低。AIGC视频生成使用本地部署的Wan2.2服务，主要成本来自GPU推理时间，系统通过分辨率、帧数、采样步数和seed参数控制生成开销。若真实上线，可迁移到PostgreSQL/MySQL、对象存储和异步任务队列。
