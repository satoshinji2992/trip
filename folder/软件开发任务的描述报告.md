## 二、项目背景

### 2.1 市场现状
当前大学生和年轻群体利用假期出行旅游已成为普遍现象。然而现有的旅游平台（如携程、马蜂窝、去哪儿等）存在以下不足：

1. **推荐算法不透明**：用户无法理解推荐逻辑，缺乏个性化调控能力
2. **景区内部导航缺失**：大部分平台仅提供城市间的交通导航，缺乏景区/校园内部的精细化路线规划
3. **路线规划策略单一**：不支持多交通工具混合、拥挤度感知等细粒度策略
4. **场所查询基于直线距离**：未考虑实际可达路径距离
5. **日记存储效率低**：大量文字内容未做压缩优化
6. **游记表达形式单一**：普通图片和文字难以形成更强的分享效果

### 2.2 项目机会
本项目旨在通过自实现核心算法（图算法、排序算法、查找算法等），构建一个**真实可用**的个性化旅游系统，解决上述痛点。

## 三、项目目标

### 3.1 核心目标
| 目标 | 说明 | 衡量标准 |
|------|------|----------|
| 数据完备 | 提供丰富的景区、校园、设施数据 | 202个景区/校园、4380个建筑、10100个设施、44440条道路边、14个用户 |
| 算法自研 | 所有核心算法基于自设计数据结构编程实现 | Dijkstra、TSP、Top-K、Trie、编辑距离、BM25F全文搜索、zlib压缩 |
| 功能完整 | 覆盖旅游前中后全流程 | 推荐、路线规划、场所查询、日记管理交流、美食推荐、室内导航 |
| 真实可用 | 系统可正常运行和演示 | 前后端完整、数据可交互、所有功能可演示 |

### 3.2 技术目标
| 目标 | 实现方式 |
|------|----------|
| 高效推荐 | Top-K排序 O(NlogK)，无需完全排序 |
| 最优路线 | Dijkstra + TSP贪心+2-opt优化 |
| 智能搜索 | Trie前缀匹配 + 编辑距离模糊查找 + Whoosh全文检索 |
| 存储优化 | zlib无损压缩日记内容 |
| 真实距离 | 基于路径图的实际可达距离（非直线距离） |
| AIGC创新 | 微调并本地部署Wan2.2图生视频大模型，生成旅游日记动画 |
| 部署能力 | Nginx + 多个Gunicorn后端实例部署，默认启动3个后端实例并通过least_conn负载均衡，发布包支持一键安装、初始化和运行 |

## 四、目标用户

### 4.1 用户画像
| 角色 | 特征 | 核心需求 |
|------|------|----------|
| 大学生 | 喜欢旅行、预算有限、时间集中在假期 | 校园游览、性价比推荐、攻略分享 |
| 旅游爱好者 | 频繁出行、关注口碑和评价 | 个性化推荐、路线规划、美食发现 |
| 背包客 | 自由行、注重路线效率 | 最短路径规划、多点路线、交通策略 |
| 美食爱好者 | 以美食为旅行动力 | 美食搜索、菜系筛选、距离排序 |

### 4.2 使用场景

**场景一：旅游前 - 选择目的地**
> 小明想利用寒假出去旅游，打开系统，根据自己的兴趣（自然风光、历史文化），系统推荐了排名前10的景区。他选择了黄山风景区，查看了详情和其他游客的日记评价。

**场景二：旅游中 - 景区内导航**
> 小红进入北京大学参观，想从正门出发，依次参观图书馆、未名湖和博物馆，最后回到正门。系统为她规划了最优游览路线，并在教学楼内提供了室内导航指引。

**场景三：旅游中 - 查找周边**
> 王五在故宫游览时想找附近的洗手间和咖啡馆，选择当前位置后，系统按实际路径距离排序列出了附近的设施。

**场景四：旅游后 - 分享日记**
> 赵六旅行归来，写下了一篇关于西湖的游记，上传了照片。其他用户可以浏览、评分和评论这篇日记。
> 如果照片质量较好，还可以选择日记中的图片，修改默认提示词和参数，生成一段适合游记展示的旅游动画。

### 4.3 用户需求满足情况

| 用户实际需求 | 系统对应功能 | 满足方式和实际价值 |
|--------------|--------------|--------------------|
| 旅游前不知道去哪里、难以比较目的地 | 景区/校园推荐、搜索、热度和评分排序 | 通过Top-K推荐和关键词查询，把用户最可能感兴趣的目的地排在前面，减少用户筛选成本 |
| 进入景区或校园后不熟悉道路 | 单点路线规划、多点路线规划、返回起点 | 使用道路图和最短路径算法给出可达路线，支持一次规划多个目标点，更符合真实游览流程 |
| 用户希望路线不仅近，还要考虑通行效率 | 最短距离、最短时间、拥挤度、交通工具策略 | 道路边保存拥挤度和交通工具限制，使系统能区分“近但拥挤”和“稍远但更快”的路线 |
| 游览过程中需要快速找到厕所、超市、餐厅等设施 | 场所查询、类别过滤、路径距离排序 | 使用Dijkstra计算实际道路距离，而不是直线距离，查询结果更接近用户真实步行体验 |
| 用户想记录和分享旅游经历 | 旅游日记、图片上传、评论评分、全文检索 | 支持文字、图片、评分和评论，并通过全文检索让其他用户能按内容发现相关游记 |
| 普通图片和文字表达形式单一 | AIGC旅游动画生成 | 通过微调并本地部署的Wan2.2图生视频模型，把日记图片生成短视频，提高游记展示效果 |
| 用户在景区/校园内需要顺便找吃的 | 美食推荐、菜系过滤、距离排序、模糊查询 | 可按热度、评分、距离和菜系筛选美食，满足游览中临时就餐和口味偏好的需求 |

## 五、项目价值

### 5.1 教学价值
| 价值点 | 说明 |
|--------|------|
| 数据结构实践 | 最小堆、最大堆、图（邻接表）、Trie树等自实现 |
| 算法实践 | Dijkstra、TSP近似、Top-K、编辑距离、BM25F等 |
| 全栈开发 | 前端React + 后端Flask + SQLite数据库 |
| 工程实践 | 模块化架构、RESTful API、前后端分离 |

### 5.2 实用价值
- 提供了一个**可运行、可演示**的完整旅游系统
- 所有核心算法均为自实现，便于理解和扩展
- 数据量满足课程要求：202个景区/校园、4380个建筑、10100个设施、44440条道路边、14个用户

### 5.3 创新点与实际价值

| 创新点 | 实现方式 | 实际价值 |
| --- | --- | --- |
| 微调并本地部署Wan2.2大模型 | 使用本地Wan2.2图生视频服务生成旅游动画 | 把普通游记图片扩展为短视频，增强日记表达和演示效果 |
| Nginx + 多后端一键部署 | 发布包默认启动3个Gunicorn后端实例，Nginx通过least_conn进行负载分发 | 降低验收和部署成本，避免单进程后端成为瓶颈，更接近真实Web系统运行方式 |
| zlib压缩存储 | 日记正文压缩后写入数据库 | 降低长文本存储占用，保证内容无损恢复 |
| BM25F全文检索 | 使用Whoosh索引日记正文、标题和目的地 | 支持按正文内容检索游记，提高查询能力 |
| 道路拥挤度设计 | 最短时间策略按拥挤度修正道路速度 | 让最短距离和最短时间路线产生可解释差异 |

## 六、项目范围

### 6.1 包含范围（In Scope）
- ✅ 用户注册、登录、个人信息管理
- ✅ 景区/校园浏览、搜索、推荐排序
- ✅ 单点最短路径和多点路线规划
- ✅ 多交通工具策略和拥挤度感知
- ✅ 室内导航（大门→电梯→楼层→房间）
- ✅ 场所查询（路径距离排序、分类过滤）
- ✅ 旅游日记CRUD、评论评分、推荐排序
- ✅ 日记全文搜索和压缩存储
- ✅ 日记图片上传、删除和AIGC旅游动画生成
- ✅ 美食推荐（菜系筛选、模糊搜索、Top-K排序）
- ✅ 地图可视化（Canvas绘制节点、边和路径）
- ✅ Gunicorn + Nginx多worker部署配置

### 6.2 不包含范围（Out of Scope）
- ⚠️ 大规模真实地图数据采集（代码支持GeoJSON/Overpass思路，当前使用故宫/北大模板复用到景区和校园）
- ⚠️ 用户原始视频上传与转码（当前支持图片上传和AIGC生成视频展示）
- ⚠️ 公网云部署和云存储（已提供Nginx/Gunicorn配置，仍需服务器、域名和对象存储账号）

## 七、风险评估

| 风险 | 等级 | 应对措施 |
|------|------|----------|
| 数据不够真实 | 中 | 使用真实景区名称和坐标，模拟内部数据 |
| 算法性能瓶颈 | 低 | Top-K避免完全排序，Dijkstra使用堆优化 |
| 前端兼容性 | 低 | 使用成熟的Ant Design组件库 |
| 数据库规模限制 | 低 | SQLite足以应对课程设计规模 |
