# 课程设计报告表内容汇总

## 1. 系统创新性和实际价值

本系统围绕“旅游前推荐、旅游中导航、旅游后记录分享”设计，创新点包括：

- 采用真实地图风格的校园/景区内部道路图，所有路线规划基于图上的可达路径，而不是直线距离。
- 路线规划加入道路拥挤度，使最短距离和最短时间可能得到不同路线，更贴近真实游览场景。
- 支持多点游览路线规划，使用Dijkstra计算关键点距离矩阵，再用贪心和2-opt优化访问顺序。
- 场所查询按实际路径距离排序，能更真实地找出附近洗手间、超市、咖啡馆等服务设施。
- 旅游日记支持图片上传、评分评论、全文检索、压缩存储，并接入Wan图生视频接口生成旅游动画。
- 推荐、美食、日记等模块使用Top-K排序，只计算用户最关注的前若干结果，避免全量排序开销。

## 2. 系统架构

系统采用前后端分离架构：

```text
React + Vite前端
  |
Axios / RESTful API
  |
Flask后端
  |
SQLAlchemy ORM
  |
SQLite数据库
```

后端按模型层、路由层、服务层组织：

- 模型层：用户、景区、建筑、道路图、设施、日记、美食等数据模型。
- 路由层：认证、旅游推荐、路线规划、场所查询、日记、美食、室内导航等API。
- 服务层：图算法、排序算法、搜索算法。
- 部署层：开发环境使用Flask和Vite；可执行包支持直接运行；生产方案可用Nginx + Gunicorn多worker。

## 5. 算法和性能分析

| 模块 | 核心算法 | 选择原因 | 复杂度 |
|------|----------|----------|--------|
| 路线规划 | Dijkstra + 最小堆 | 适合非负权道路图，能同时支持距离和时间权重 | O((V+E)logV) |
| 多点规划 | Dijkstra矩阵 + 最近邻 + 2-opt | 精确TSP代价高，近似算法更适合演示和实时交互 | 约O(K·(V+E)logV + K²·I) |
| 推荐排序 | Top-K最小堆 | 用户通常只看前10，避免全量排序 | O(NlogK) |
| 模糊查询 | 编辑距离 + 分词匹配 | 能处理用户输入不完整或有错字的情况 | O(MN) |
| 名称查询 | Trie前缀树 | 适合高频前缀查找 | O(L) |
| 全文检索 | BM25F | 比简单关键词包含更适合长文本相关性排序 | 由Whoosh索引加速 |
| 场所距离排序 | Dijkstra + 距离排序 | 使用真实路径距离，符合课程要求 | O((V+E)logV + MlogM) |
| 日记压缩 | zlib无损压缩 | 降低长文本存储空间，不损失内容 | 与文本长度线性相关 |

## 6. AI开发模式总结

开发中采用AI Coding辅助完成功能拆解、代码实现、问题定位和文档整理。典型协作方式：

- 先由人提出课程验收要求和演示目标，AI帮助拆解为数据、功能、算法、文档四类任务。
- 人在浏览器中验证功能现象，如路线失败、搜索异常、上传体验问题；AI根据现象定位前后端代码。
- AI辅助补齐初始化数据、接口字段、前端交互和答辩材料，人负责确认演示效果和需求优先级。
- 对算法部分，AI辅助整理复杂度、适用场景和答辩话术，保证能讲清楚“为什么这样设计”。

这种模式提高了迭代速度，但也要求开发者能判断AI输出是否符合课程要求，尤其是数据规模、算法口径和真实运行效果。

## 7. 实践反思

本次课程设计让我更清楚地认识到，AI可以加速编码和文档生成，但不能代替对系统整体的理解。比如路线规划失败、场所距离排序反向、AIGC接口参数不兼容等问题，都需要结合数据库、后端算法和前端表现一起排查。后续改进方向包括接入更多真实地图数据、支持更完善的并发测试、进一步优化多点路线规划和全文检索体验。

## 8. 系统性能和成本

- 数据库：SQLite适合课程设计和单机演示，部署简单、成本低；生产环境可迁移到PostgreSQL/MySQL。
- 并发：开发环境使用Flask单进程；部署方案使用Nginx + Gunicorn多worker，提高并发处理能力。
- 算法：推荐和排序使用Top-K降低排序成本；路线规划使用堆优化Dijkstra，适合当前每个地图220条边的规模。
- 存储：日记正文使用zlib压缩；图片和视频保存在本地上传目录，后续可迁移到对象存储。
- AIGC成本：视频生成依赖外部Wan服务，主要成本来自GPU推理时间；系统通过可调分辨率、帧数、采样步数控制成本。
