## 四、核心算法详解

### 4.1 Dijkstra最短路径算法 (`graph_service.py`)

- **数据结构**: 自实现MinHeap（最小堆）作为优先队列
- **时间复杂度**: O((V+E)logV)
- **支持**: 最短距离和最短时间两种权重
- **交通工具适配**: 根据交通模式过滤可通行的边，调整速度

### 4.2 多点最短路径TSP近似 (`graph_service.py`)

- **步骤1**: 计算所有关键点之间的最短距离矩阵（调用Dijkstra）
- **步骤2**: 贪心算法构建初始解（最近邻法）
- **步骤3**: 2-opt局部优化（反转子路径减少总距离）
- **支持**: 是否返回起点选项

### 4.3 Top-K排序算法 (`sort_service.py`)

- **数据结构**: 自实现MinHeapForTopK（容量为K的最小堆）
- **时间复杂度**: O(N·logK)，优于完全排序的O(N·logN)
- **应用**: 景区推荐、日记推荐、美食推荐——只需排好前10个

### 4.4 推荐评分算法 (`sort_service.py`)

综合评分公式：`score = 0.4 × 热度归一化 + 0.3 × 评价归一化 + 0.3 × 兴趣匹配度`

### 4.5 模糊查找算法 (`search_service.py`)

- **编辑距离**: 自实现Levenshtein距离DP算法
- **匹配策略**: 完全包含 > 分词匹配 > 子串模糊匹配
- **Trie树**: 自实现字典树，用于高效前缀查找

### 4.6 全文搜索 (`search_service.py`)

- 基于Whoosh库实现BM25F评分
- 支持标题、内容、目的地、标签多字段检索

### 4.7 无损压缩存储 (`diary.py`)

- 使用zlib level=9压缩日记内容
- `set_content()` 压缩存储，`get_content()` 解压读取
- API返回压缩率信息

### 4.8 AIGC旅游动画生成 (`diary.py`)

- 选择日记内任意图片作为输入
- 默认提示词可由前端展示并允许用户修改
- 支持横屏/竖屏、480p/720p、秒数、采样步数和seed参数
- 后端按Wan接口格式提交 `image`、`prompt`、`size`、`sample_steps`、`frame_num`、`seed`
- 生成结果保存到 `backend/uploads/diary/{diary_id}/animation_*.mp4`
- 日记详情接口扫描该目录下的 `animation_*.mp4`，以 `videos` 字段返回给前端展示

### 4.9 路径距离排序 (`sort_service.py`)

- 一次Dijkstra计算从源点到所有节点的最短距离
- 按**实际可达路径距离**排序（非直线距离）

### 5.4 Nginx + 多后端实例部署

开发环境可使用 Flask 内置服务；部署环境使用 Nginx 反向代理到多个 Gunicorn 后端实例，以提升并发处理能力和运行稳定性。默认会同时启动 3 个后端实例：

```text
127.0.0.1:5001
127.0.0.1:5002
127.0.0.1:5003
```

每个后端实例内部仍由 Gunicorn 启动多个 worker/线程处理 Flask 请求；Nginx 则负责在多个后端实例之间做负载分发。启动脚本和配置文件位于：

```text
setup_and_start_nginx.sh
deploy/nginx.trip.conf
```

架构：

```text
浏览器
  -> Nginx 多 worker
  -> least_conn 负载均衡
  -> Gunicorn 后端实例 5001 / 5002 / 5003
  -> Flask API
```

Nginx 负责静态资源、反向代理和并发连接，并通过 `least_conn` 把 `/api/` 和 `/uploads/` 请求转发给连接数较少的后端实例。这样不是只开一个后端进程，而是“多个后端实例 + 每个实例多个 worker”的结构，能避免单进程 Flask 开发服务器成为瓶颈。后端实例数量可以通过环境变量调整，例如：

```bash
BACKEND_INSTANCES=4 ./setup_and_start_nginx.sh
```

## 六、拥挤度和交通工具策略

### 6.1 拥挤度策略
- 每条道路边有 `congestion` 属性（0~1）
- 实际速度 = max(0.1, 1-congestion) × 基础速度
- 最短时间策略会自动考虑拥挤度
- 初始化脚本会写入确定性的拥挤度分布：主路和热门短连线更拥挤，支路更通畅
- 因此同一组起终点下，“最短距离”可能选择近但堵的路，“最短时间”可能选择绕行但畅通的路

### 6.2 交通工具策略
- **校区内**: 步行(5km/h) + 自行车(15km/h)，自行车任何地点都有
- **景区内**: 步行(5km/h) + 电瓶车(20km/h)，电瓶车路线固定
- 每条边有 `transport_allowed` 属性限制可通行的交通工具

## 四、路线规划接口 (`/api/route`)

### POST `/api/route/shortest` - 最短路径（Dijkstra）
**请求体**:
```json
{
  "scenic_id": 1,
  "from_node_id": 1,
  "to_node_id": 10,
  "strategy": "distance",
  "transport": "walk"
}
```
- `strategy`: `distance`(最短距离) / `time`(最短时间)
- `transport`: `walk`(步行) / `bike`(自行车) / `cart`(电瓶车)

**响应**:
```json
{
  "path": [1, 3, 5, 10],
  "path_details": [{ "node_id": 1, "name": "正门", "x": 100, "y": 200 }, ...],
  "total_cost": 450.5,
  "cost_unit": "米",
```

# Nginx 多 Worker 部署说明

## 1. 部署目标

开发环境中可以直接使用 Flask 内置服务器和 Vite 开发服务器；部署环境采用 Nginx + 多个 Gunicorn 后端实例的方式承载并发访问：

```text
浏览器 -> Nginx 多 worker -> 多个 Gunicorn 后端实例 -> Flask 后端 -> SQLite
```

其中：

- Nginx 负责静态资源、反向代理、请求转发、上传大小限制。
- Gunicorn 负责启动多个后端 worker，提高并发处理能力。
- Flask 只负责业务接口和算法计算。

## 2. 前端构建

```bash
cd /Users/shenqi/Documents/code/trip/frontend
npm install
npm run build
```

构建产物位于：

```text
frontend/dist
```

Nginx 会直接托管这个目录。

## 3. 后端多 Worker 启动

发布包推荐使用 `setup_and_start_nginx.sh` 自动启动多个后端实例。默认启动 3 个 Gunicorn 实例：

```text
127.0.0.1:5001
127.0.0.1:5002
127.0.0.1:5003
```

也可以手动启动：

```bash
cd /Users/shenqi/Documents/code/trip/backend
pip install -r requirements.txt
GUNICORN_BIND=127.0.0.1:5001 gunicorn -c ../deploy/gunicorn.conf.py run:app
GUNICORN_BIND=127.0.0.1:5002 gunicorn -c ../deploy/gunicorn.conf.py run:app
GUNICORN_BIND=127.0.0.1:5003 gunicorn -c ../deploy/gunicorn.conf.py run:app
```

默认 worker 数：

```text
CPU核数 * 2 + 1
```

也可以手动指定：

```bash
GUNICORN_WORKERS=4 GUNICORN_THREADS=2 gunicorn -c ../deploy/gunicorn.conf.py run:app
```

## 4. Nginx 启动

配置文件：

```text
deploy/nginx.trip.conf
```

本机测试可以使用：

```bash
nginx -c /Users/shenqi/Documents/code/trip/deploy/nginx.trip.conf
```

如果已有 Nginx，可把 `server` 部分合并到系统 Nginx 配置中，并把 `root` 改成实际的 `frontend/dist` 路径。

## 5. Nginx 配置重点

```nginx
worker_processes auto;

events {
    worker_connections 1024;
    multi_accept on;
}
```

含义：·

- `worker_processes auto`：根据 CPU 自动开启多个 Nginx worker。
- `worker_connections 1024`：每个 worker 最多处理 1024 个连接。
- `multi_accept on`：worker 可以一次接受多个新连接。

后端 upstream：

```nginx
upstream trip_backend {
    least_conn;
    server 127.0.0.1:5001;
    server 127.0.0.1:5002;
    server 127.0.0.1:5003;
}
```

含义：

- Nginx 将 `/api/` 和 `/uploads/` 转发给多个 Gunicorn 后端实例。
- `least_conn` 表示优先转发给连接数较少的后端实例，实现单机多实例负载分发。

## 6. 部署性能指标说明

| 指标         | 说明                                               |
| ------------ | -------------------------------------------------- |
| 并发连接     | Nginx worker 数 × worker_connections               |
| 后端并发     | Gunicorn workers × threads                         |
| 静态资源性能 | 前端 dist 由 Nginx 直接返回，减少 Flask 压力       |
| 大文件上传   | `client_max_body_size 200m` 支持日记图片和视频结果 |
| 长耗时接口   | `proxy_read_timeout 300s` 支持 AIGC 视频生成等待   |

## 7. 部署注意事项

- SQLite 适合轻量部署和单机运行；如果并发写入明显增加，建议迁移到 PostgreSQL 或 MySQL。
- AIGC 视频生成耗时较长，生产环境可以进一步改成异步任务队列。
- 如果部署到服务器，需要把 `server_name` 改成实际域名，并配置 HTTPS。
